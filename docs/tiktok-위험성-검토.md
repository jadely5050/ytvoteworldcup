# 틱톡(TikTok) 연동 — 위험성 및 검토 결과 정리

> 작성일: 2026-06-20 · 대상: ytvoteworldcup 라이브 투표 오버레이
> 한 줄 요약: **틱톡 라이브 채팅 수집은 기술적으로 가능하지만, 외부 사인서버 의존·적극적 차단·약관 위반 소지로 위험도가 치지직/유튜브보다 높다. 현재는 미구현(검토만) 상태이며 "되면 보너스" 수준으로 권장.**

---

## 1. 현재 상황 (Status)

| 항목 | 상태 |
|---|---|
| 구현 | ❌ 미구현 (검토만 완료) |
| 권장 사용 라이브러리 | `tiktok-live-connector` (npm, zerodytrash) |
| 인증/키 | 기본 익명 읽기는 가능하나, 안정 연결에 **외부 사인서버(EulerStream)** 의존 추세 |
| 공식 API로 실시간 댓글 | ❌ 불가 (Research API는 승인 연구자·사후 데이터용, 실시간 라이브 채팅 아님) |
| 결정 | 이번 범위에서 **제외**. 유튜브+치지직으로 운영 |

### 작동 방식 (구현 시)
- 틱톡은 라이브 채팅 공식 실시간 API가 없어, **틱톡 앱 내부 통신(Webcast 푸시 서비스)을 리버스 엔지니어링**해 흉내 낸다.
- 로그인 없이 남의 라이브 채팅을 익명으로 듣는 방식 — `new TikTokLiveConnection('@username')` → `.on('chat', ...)`.
- 최근 틱톡이 연결 요청에 **암호 서명**을 요구하면서, 안정적 연결을 위해 **제3자 사인 서버(EulerStream)**를 한 번 더 경유하는 구조가 됐다.

---

## 2. 위험성 분석 (Risk Assessment)

### (가) 비공식 API + 적극적 차단 → 라이브러리 붕괴 위험 · 높음
- 유튜브·치지직도 비공식이지만, **틱톡은 외부 접근을 의도적으로·적극적으로 막는다.**
- 틱톡이 내부 통신/보안 서명을 바꾸면 라이브러리가 **하루아침에 먹통**이 되고, 패치까지 며칠~몇 주 걸릴 수 있다.
- 하필 중요한 생방송 날 터지면 그날 틱톡 투표가 죽는다. → **안정성이 가장 큰 약점.**

### (나) 계정 밴 위험 · 낮음(읽기 전용 시) / 높음(로그인 시)
- **로그인 없이 읽기만** 하면 밴당할 '내 계정'이 없어 위험이 낮다.
- 단, 채팅 전송·하트 등 **인증이 필요한 기능을 쓰려고 계정을 연결하면** 그 계정은 약관 위반으로 **제재(밴) 위험**이 있다. → 우리 용도(읽기)에선 로그인하지 말 것.

### (다) 차단(IP/연결) 위험 · 중간~높음
- 틱톡이 "정상 앱이 아닌 봇"으로 판단하면 **IP·연결을 끊는다.** 방송 도중 수집이 멈출 수 있다.
- 치지직보다 탐지·차단이 공격적이라는 평가가 많다.

### (라) 외부 사인서버(EulerStream) 의존 · 중간
- 우리 서버가 틱톡에 직접 붙는 게 아니라 **외부 업체를 한 번 더 경유**해야 안정적으로 붙는다.
- 무료 등급은 **요청 횟수 제한**, 많이 쓰면 **유료**. 그 외부 서비스가 죽거나 정책을 바꾸면 우리도 함께 영향을 받는다.
- 유튜브·치지직엔 없는 **추가 의존성·비용 리스크.**

### (마) 약관(ToS) 위반 성격 · 명확한 회색~위반
- 라이브러리 문서 자체가 "production-ready 아님 / 약관 위반 소지 / 계정 밴 경고"를 명시한다.
- 법적으로 잡혀가는 류라기보다, **서비스 약관 위반 + 차단 위험**의 성격이다.

## 3. 플랫폼 비교

| | 유튜브 | 치지직 | **틱톡** |
|---|---|---|---|
| 방식 | 비공식(masterchat) | 비공식(chzzk) | **비공식 + 외부 사인서버** |
| 인증/키 | 영상 ID만 | 채널 ID만 | **사인서버 의존** |
| 계정 밴 위험 | 없음(익명) | 없음(익명) | 없음(읽기) / 높음(로그인) |
| 갑자기 끊길 위험 | 낮음 | 낮음~중간 | **높음** |
| 외부 유료 의존 | 없음 | 없음 | **있을 수 있음** |
| 실시간 공식 API | ❌ | 송신 위주 | ❌(연구용·실시간 아님) |
| 종합 위험도 | 낮음 | 낮음~중간 | **중간~높음** |

## 4. 만약 구현한다면 (권장 조건)

틱톡을 넣더라도 다음 전제를 지키면 피해를 최소화할 수 있다.

1. **읽기 전용** — 로그인하지 않고 채팅 수신만. (계정 밴 위험 회피)
2. **소스 독립 구조** — 현재 코드처럼 틱톡이 끊겨도 유튜브·치지직 투표는 정상 동작하게. 한 플랫폼 장애가 전체로 번지지 않도록.
3. **"보너스" 취급** — 메인 집계는 유튜브+치지직, 틱톡은 되면 더해지는 정도. 중요한 방송에서 틱톡 단독에 의존하지 말 것.
4. **사인서버 정책 확인** — EulerStream 무료 한도/유료 조건을 사전 점검.
5. **방송 전 리허설 필수** — 틱톡은 그날 갑자기 안 될 수 있으므로 직전 동작 확인.

> 구현 난이도 자체는 치지직과 비슷(어댑터 패턴: `sources/tiktok.js` 추가). 어려운 건 코드가 아니라 **운영 안정성**이다.

---

## 5. 결론

- 틱톡은 **"계정이 밴된다"가 핵심 위험이 아니라(읽기만 하면 거의 없음), "믿고 생방송에 올렸다가 그날 안 터질지 장담 못 한다"는 안정성·운영 리스크**가 핵심이다.
- 외부 사인서버 의존과 적극적 차단 때문에 **유튜브·치지직보다 한 단계 더 불안정**하다.
- 따라서 권장은 명확하다: **유튜브+치지직을 메인으로 운영하고, 틱톡은 읽기 전용·독립 구조 전제의 보너스로만** 접근한다. 필요해지면 `sources/tiktok.js`로 동일 어댑터 패턴 확장이 가능하다.
